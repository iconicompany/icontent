# Разработка на основе спецификаций (SDD) и ИИ-агенты: Полное руководство по выходу из петли «вайб-кодинга»
Профессия разработчика переживает самый тектонический сдвиг со времен появления языков высокого уровня. ИИ-агенты — Claude Code, GitHub Copilot, Cursor, Windsurf — научились писать, рефакторить и тестировать код со скоростью пулемета. Но сырая скорость без точности — это не актив. Это технический долг, умноженный на вычислительную мощность.
Сегодня большинство команд используют ИИ-агентов вслепую. Они пишут промпт, получают оверлей, латают возникшие ошибки и повторяют это по кругу. По мере роста кодовой базы этот подход неизбежно заходит в тупик. Причина не в «глупости» нейросети, а в отсутствии дисциплинированного фреймворка для передачи контекста и намерений.
**Spec-Driven Development (SDD) — Разработка на основе спецификаций** — это инженерная дисциплина, в которой структурированная спецификация становится *главным артефактом* разработки. Это суверенный документ, из которого деривативно выводится, проверяется и управляется вся реализация.
## Часть 1. Концептуальный фундамент
### 1.1. Крах «вайб-кодинга»
**Вайб-кодинг (Vibe coding)** — практика, когда разработчик описывает желаемый результат разговорным языком и итерирует код через неформальные промпты. Это круто работает для изолированных скриптов, быстрых MVP и pet-проектов на один вечер. Но на масштабе энтерпрайза этот подход терпит катастрофический крах. И вот почему.
#### Деградация контекста (Context Decay)
Любая сессия с ИИ-агентом ограничена контекстным окном. В режиме вайб-кодинга состояние проекта «размазано» по истории диалога. По мере роста чата ранние архитектурные решения, требования к безопасности и ограничения библиотек банально вылетают из памяти модели. Агент начинает генерировать правдоподобные, но ошибочные альтернативы, даже не предупреждая вас.
Деградация контекста приводит к **дрейфу реализации (implementation drift)** — постепенному расхождению между тем, как систему понимает агент, и реальными требованиями бизнеса. Код может идеально проходить тесты и работать изолированно, но при этом нарушать глобальные архитектурные или бизнес-правила. В энтерпрайзе диагностика и исправление таких дефектов обходятся дороже всего.
### 1.2. Базовые принципы SDD
В SDD спецификация — это первичный артефакт, а код — вторичный (производный) результат. Это инверсия классического подхода, где код первичен, а документация пишется постфактум (или не пишется вообще).
В рамках SDD спецификация должна быть:
 * Машиночитаемой;
 * Версионируемой (лежать в Git);
 * «Живой» и исполняемой.
Она выступает в роли жесткого контракта между человеком и агентом. Агент не должен додумывать или интерпретировать — он реализует код строго в рамках контракта. При точной спецификации на выходе получается предсказуемый результат. При двусмысленной — колоссальная вариативность.
> **Главный инсайт SDD:** Вы не можете контролировать то, что агент строит, если вы точно не определили, *что именно* должно быть построено. Точность спецификации — это единственный рычаг управления качеством кода.
> 
### 1.3. Шесть элементов машиночитаемой спецификации
Чтобы агент не делал ложных допущений, спецификация должна содержать 6 обязательных блоков:
 1. **Бизнес-цели и результаты:** Зачем мы это пишем и какую проблему решаем.
 2. **Границы скоупа (Scope Boundaries):** Четко очерченные рамки — что входит в задачу, а что категорически нет.
 3. **Технические ограничения:** Стек, запрещенные библиотеки, требования к производительности.
 4. **Ключевые архитектурные решения (ADR):** Почему выбран именно этот паттерн или структура данных.
 5. **Пошаговый план декомпозиции:** Список атомарных задач для агента.
 6. **Критерии приемки и верификации:** Как автоматически проверить, что задача выполнена успешно.
## Часть 2. Рабочий процесс (Workflow) в SDD
### 2.1. Четырехфазная модель
Процесс SDD разбит на 4 последовательные фазы. На стыке каждой из них находится **Human Intervention Point (HIP)** — точка обязательного контроля человеком. Это не бюрократия, а предохранительные шлюзы. Попытка проскочить HIP ради скорости всегда приводит к обратному результату.
```
[ Фаза 1: Спецификация ] ---> (HIP 1: Ревью спеки человеком)
           |
           v
[ Фаза 2: Проектирование ] -> (HIP 2: Ревью архитектуры человеком)
           |
           v
[ Фаза 3: Декомпозиция ] ----> (HIP 3: Валидация атомарных тасок)
           |
           v
[ Фаза 4: Кодинг и Верификация ] -> (Проверка агентом-верификатором)

```
 * **Фаза 1: Спецификация (Specify).** Разработчик описывает верхнеуровневое намерение. Агент (или сам разработчик) разворачивает его в структурированную функциональную спецификацию. Роль человека здесь — архитектурный надзор: отловить галлюцинации и упущения еще *до* планирования реализации. Если в спеке осталась двусмысленность — дальше идти нельзя. Агент выдаст не сомневающийся код, а уверенный в себе, но абсолютно неверный.
 * **Фаза 2: Проектирование (Plan).** На основе спеки агент создает технический план: схемы данных, контракты API, выбор конкретных технологий. Здесь закладываются корпоративные ограничения (например, *"использовать только PostgreSQL"* или *"все эндпоинты закрыты JWT"*). Человек валидирует этот план, пресекая архитектурные ошибки на корню.
 * **Фаза 3: Декомпозиция (Task Breakdown).** План разбивается на список атомарных, независимо тестируемых задач. Размер задачи — критически важная переменная. Каждая таска должна выполняться и проверяться за *один* цикл итерации агента. Если задача требует от ИИ удерживать в памяти более 3 независимых контекстов одновременно — дробите её.
 * **Фаза 4: Реализация и Верификация (Implement & Verify).** Агент пишет код строго по таскам. Этот этап обязательно завязан на **паттерн Верификатора (Verifier pattern)**: один агент генерирует код, а второй (или автоматизированный тест-сьют) проверяет результат на соответствие критериям приемки.
### 2.2. Двунаправленное обновление спецификации (Bidirectional Updates)
Самое частое нарушение в SDD: агент в процессе кодинга принимает локальное решение (например, выбрал конкретную библиотеку для оптимизации или неочевидным образом обработал эдж-кейс), но спецификация об этом «не знает».
Возникает **долг спецификации (specification debt)**. Спека превращается в неактуальный хлам, которому никто не верит. Правило должно быть жестким и непускающим компромиссов:
> Если реализация отклонилась от спецификации — обновите спецификацию. Не завтра. Не в следующем спринте. Прямо сейчас. Долг спецификации накапливается быстрее технического долга, потому что он отравляет входные данные для всех последующих генераций ИИ.
> 
## Часть 3. Артефакты персистентности и Конституция проекта
ИИ-агент не имеет памяти между сессиями. Каждый новый чат — это чистый лист (tabula rasa). Без формального механизма персистентности (сохранения контекста) агент понятия не имеет о ваших стандартах безопасности, кодстайле, запрещенных пакетах и прошлых архитектурных договоренностях.
Для решения этой проблемы в корень репозитория коммитится **Конституция проекта** — набор структурированных .md файлов.
### Основные артефакты:
 1. AGENTS.md / CLAUDE.md — **Операционная инструкция.** Это README для ИИ-агента, который он обязан прочесть на старте сессии. Файл пишется для машины, а не для человека, и содержит:
   * Команды сборки, запуска тестов, линтинга со всеми флагами.
   * Конвенции по именованию файлов, каталогов и модулей.
   * **Иерархию строгости действий:**
     * *Всегда выполнять самостоятельно:* линтинг, запуск затрагиваемых тестов.
     * *Спросить человека перед выполнением:* изменение схемы БД, добавление новых зависимостей, правка контрактов API.
     * *Категорически запрещено:* коммитить секреты, править файлы .env, менять логику авторизации без ручного апрува.
 2. SPEC.md — **Живая спецификация.** Актуальное описание того, что система делает прямо сейчас. Меняются требования — сначала правится SPEC.md, а затем код.
 3. PLAN.md — **Журнал архитектурных решений (ADR).** Фиксирует, *почему* система спроектирована именно так, защищая кодовую базу от того, что пришедший завтра новый ИИ-агент (или джун) перепишет оптимальное решение на костыль.
> **Важно:** Обратите внимание на формат конфигурационных файлов .mdc в Cursor. Они позволяют привязывать правила к конкретным маскам файлов (glob patterns). Правила бэкенда (авторизация, SQL-запросы) будут активны только тогда, когда агент работает в папке /backend, не забивая контекстное окно во время верстки фронтенда.
> 
## Часть 4. Мультиагентная оркестровка
Один агент физически не способен вытащить энтерпрайз-проект на 100 000+ файлов из-за лимитов контекстного окна. Ему приходится выбирать, что читать, и в этих выборах появляются слепые зоны. Выход — мультиагентная архитектура, повторяющая структуру зрелой инженерной команды.
### Архитектура трех ролей:
 * **Агент-Координатор (Coordinator):** Выступает в роли ПМа и системного архитектора. Он видит картину целиком: глобальную спеку, глобальный план и граф задач. Его работа — декомпозиция и делегирование (разбить спеку на суб-спеки и раздать их Исполнителям). Он следит за тем, чтобы параллельные стримы не конфликтовали между собой.
 * **Агент-Исполнитель (Implementor):** Узкоспециализированный агент. Получает изолированную подзадачу и работает в **отдельном Git-ворктрей (worktree)**. Он ничего не знает о соседних задачах, и эта изоляция обеспечивает детерминированность и чистоту его кода.
 * **Агент-Верификатор (Verifier):** У него противоположная Исполнителю цель. Если задача Исполнителя — написать код, то задача Верификатора — сломать его и найти несоответствия спецификации. Это классический состязательный (adversarial) паттерн. Верификатор ищет зазоры в логике, нарушения безопасности и дрейф намерений. Если найдена ошибка, он не правит код сам, а возвращает Координатору структурированный баг-репорт со ссылкой на пункт в SPEC.md.
## Часть 5. Diagrams as Code: Сначала схемы, потом код
Архитектурные диаграммы, создаваемые постфактум, бесполезны. В SDD диаграммы — это инструмент проектирования «до» написания кода.
Требование к агенту сгенерировать валидную **Mermaid** или **D2** диаграмму заставляет его явно просчитать структурные связи, потоки данных и переходы состояний. Если диаграмма на выходе получается запутанной или ломается — это явный маркер того, что спецификация содержит логические дыры. Исправить схему в текстовом формате Mermaid в разы дешевле, чем переписывать готовые микросервисы.
## Часть 6. Три уровня зрелости SDD
Команды внедряют SDD на разных уровнях в зависимости от потребностей проекта.
| Уровень | Подход | Плюсы / Минусы | Когда использовать |
|---|---|---|---|
| **Level 1: Spec-First** | Спека пишется перед задачей, но после верификации кода выбрасывается или не обновляется. | **+** Быстрый старт, минимум оверхеда.
**-** Контекст быстро теряется при долгой поддержке. | Изолированные фичи, легаси-миграции, разовые эксперименты. |
| **Level 2: Spec-Anchored** | Спецификация и код — равноправные партнеры. Любые изменения синхронизируются в обе стороны. | **+** Идеальное сохранение контекста, контролируемый техдолг.
**-** Кажется, что есть нагрузка на «двойное ведение». | **Рекомендуемый стандарт** для продуктовой разработки и долгоживущих систем. |
| **Level 3: Spec-as-Source** | Спека — единственный исходный код. Сам код проекта — это просто компилируемый ИИ-агентом артефакт. | **+** Максимальная автоматизация.
**-** Нет детерминизма. Git-диффы и отладка превращаются в кошмар. | Экспериментальные гринфилд-проекты с жестким ИИ-гавернансом. |
## Часть 7. Шаблон AGENTS.md для вашего репозитория
Минимально жизнеспособный шаблон, который можно положить в корень проекта уже сегодня:
```markdown
# SYSTEM PROMPT & OPERATIONAL CONTEXT FOR AI AGENTS

## 1. Project Identity & Stack
- Core: Node.js v22, TypeScript v5, Next.js v15 (App Router)
- Database: PostgreSQL via Prisma ORM
- Testing: Jest + Playwright

## 2. Structural Conventions
- All API routes strictly live in `/src/app/api/`
- Shared UI components must be placed in `/src/components/ui/` and use Tailwind CSS
- Business logic must be decoupled from components and reside in `/src/lib/services/`

## 3. Agent Execution Rules & Severity Hierarchy
### ALWAYS DO (No Permission Required):
- Run `npm run lint` and `npm run test` on affected files before declaring a task done.
- Follow existing naming conventions (camelCase for files, PascalCase for components).

### ASK BEFORE DOING:
- Adding any external npm package.
- Modifying `prisma.schema` or creating DB migrations.
- Changing shared Tailwind or Typescript configurations.

### NEVER DO:
- Hardcode secrets, API keys, or tokens (use `process.env` exclusively).
- Commit code that breaks the main build or introduces TypeScript compilation errors.
- Modify `.github/workflows/` or CI/CD pipelines without explicit human review.

## 4. Verification Workflow
Every code change must be accompanied by a corresponding unit test in the `.test.ts` file covering primary success paths and at least two edge cases.

```
## Вместо итога: Спецификация и есть продукт
Эра ИИ-агентов четко дала понять: бутылочное горлышко в разработке ПО сместилось. Проблема больше не в скорости написания синтаксиса — машина пишет его мгновенно. Проблема в **точности определения намерений**.
Код становится лишь побочным продуктом компиляции ваших мыслей. В агентской реальности суверенная, детальная спецификация — это самый главный и ценный результат работы инженера.
